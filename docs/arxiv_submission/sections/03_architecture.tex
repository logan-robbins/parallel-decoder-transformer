\section{Architecture and Curriculum}
\label{sec:architecture}

The Parallel Decoder Transformer (PDT) is a decoder-only architecture that turns one frozen backbone into multiple coordinated generation streams. The key design decision is that decomposition and cross-stream state are model-internal: the prompt is planned once, the resulting latent plan seeds a shared bus, and streams decode against that bus rather than against separate prompt strings.

\subsection{The Frozen Trunk Topology}
We initialize PDT with a pre-trained GPT-OSS backbone parameterized by $\theta_{\text{pre}}$. We freeze all weights in $\theta_{\text{pre}}$. To enable task-specific behavior, we inject a lightweight set of trainable parameters $\phi$.
The total parameters $\Theta = \theta_{\text{pre}} \cup \phi$. The trainable set $\phi$ consists of:
\begin{itemize}
    \item \textbf{Stream Adapters:} Bottleneck MLPs inserted into selected transformer blocks to inject stream-specific conditioning.
    \item \textbf{SNC Backends:} Cross-attention layers (Section \ref{sec:snc}) that read from the shared Dynamic Notes Bus.
    \item \textbf{Planner and Notes Modules:} A fixed-slot \texttt{planner\_head}, a shared \texttt{plan\_embedding}, and a projection layer that maps latent plan IDs into the notes space used by the bus.
    \item \textbf{Auxiliary Heads:} Lightweight heads for note prediction, speculative note emission, coverage prediction, agreement scoring, and stream classification.
\end{itemize}

\subsection{Prompt-Time Latent Planner}

PDT uses a mandatory planner pass before token decoding begins. The frozen trunk first encodes the prompt into hidden states $\mathbf{H}_x \in \mathbb{R}^{T \times d}$. The planner then pools those prompt states and predicts a fixed set of latent plan slots:
\begin{equation}
    \mathbf{P} = \mathrm{PlannerHead}(\mathbf{H}_x) \in \mathbb{R}^{S \times V_p}, \qquad
    \hat{z}_i = \arg\max_{v} \mathbf{P}_{i,v},
\end{equation}
where the current implementation uses $S=16$ planner slots and a shared latent plan vocabulary of size $V_p = 65{,}536$.

These slot IDs are \emph{not} LM tokens. They live in the same latent vocabulary that also indexes \texttt{plan\_embedding}, canonical plan catalog hashes, and coverage targets. Each active slot is re-embedded and projected into notes space:
\begin{equation}
    \mathbf{e}_i = E_{\text{plan}}[\hat{z}_i], \qquad
    \mathbf{n}^{\text{plan}}_0 = \mathrm{Pool}\!\left(W_{\text{plan}} \mathbf{e}_{1:S}\right).
\end{equation}
The pooled plan seed $\mathbf{n}^{\text{plan}}_0$ is broadcast to every stream as snapshot 0 on the Dynamic Notes Bus.

The planner head solves the initial latent code-selection problem. Downstream, \texttt{plan\_embedding} and \texttt{plan\_notes\_proj} convert those latent IDs into the shared snapshot-0 seed, SNC consumes the visible workspace on each decode step, \texttt{notes\_head} writes refreshed summaries back to the bus, and coverage/agreement heads regulate overlap and rollback.

\subsection{Parallel Streams and Dynamic Notes Bus}

Instead of a single causal stream, PDT maintains $K$ parallel streams. All streams share the same frozen trunk parameters $\theta_{\text{pre}}$ but maintain distinct KV-caches, adapter states, and decode positions. Coordination is mediated by the \textbf{Dynamic Notes Bus}, a versioned snapshot store populated with fixed-dimensional note embeddings.

The bus carries embeddings only. Textual plans, note renderings, and coverage strings are auxiliary artifacts used for dataset construction, observability, and scoring. At decode step $t$, every stream reads the currently visible lagged notes window; cadence and agreement logic determine whether a new summary is written back to the bus or whether a rollback event is triggered. The runtime uses an all-to-all broadcast topology over snapshot windows, so a stream can condition on the latest visible summaries from its siblings without exchanging raw text.

Snapshot 0 is part of the data and training contract. The dataset materializes \texttt{versioned\_notes[0]} from \texttt{plan\_contract}, and training preserves that initial snapshot through the first stride so the planner seed remains stable during early decode steps.
Because canonical plan items are stream-qualified, snapshot 0 also encodes an initial ownership structure. Coverage and rollback heads therefore judge not just whether content was produced, but whether the shared latent plan is being executed by the intended stream without unnecessary overlap.

\subsection{Parameter-Efficient Curriculum}
\label{sec:curriculum}
Training a parallel coordination mechanism on a frozen trunk is unstable if attempted end-to-end. We therefore use a staged curriculum that progressively enables the planner, bus, and rollback machinery.
\begin{itemize}
    \item \textbf{Stage 0 (Planner Pretrain):} Train \texttt{planner\_head}, \texttt{notes\_head}, and \texttt{plan\_notes\_proj} while the trunk remains frozen. Supervision comes from latent \texttt{planner\_ids} and the snapshot-0 plan contract.
    \item \textbf{Stage 1 (Stream Bootstrap):} Unfreeze stream adapters and SNC cross-attention so streams learn stream-specific conditioning against fixed dataset snapshots while teacher notes remain the supervision target.
    \item \textbf{Stage 2 (Notes Bus Enable):} Unfreeze \texttt{speculation\_head}. Streams begin emitting learned note embeddings into the versioned bus while the planner seed remains fixed at snapshot 0.
    \item \textbf{Stage 3 (Rollback Training):} Unfreeze \texttt{coverage\_head} and \texttt{agreement\_head}. Coverage is predicted against canonical plan catalog items, and agreement supervision controls rollback behavior over the commit horizon.
\end{itemize}
This curriculum keeps the planner-first path canonical throughout training: latent planning seeds the bus first, stream adapters and SNC consume that seed, note-emission heads refresh the workspace, and later stages add item-level coverage and rollback control without changing the representational contract.
